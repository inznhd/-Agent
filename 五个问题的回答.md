# **模块一：Context / Performance**

### 一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

1. 短期（context）压缩：给llm提供完整的历史记录，并提供一份压缩总结模板，模板里包含所有我们需要的关键内容（背景信息、项目目标、当前进度/执行历程、用户约束等）。把压缩内容替换原有上下文时，保留原上下文最新的几轮回复，确保最新的对话连贯流畅。

2. 压缩的时候可以同时做项目记忆、长期记忆、用户个性化内容等的总结和存储；双向的，把项目记忆插入压缩后的上下文。更多细节属于memory管理模块了。

3. 最后我认为agent针对他服务面向不同的垂直领域进行优化很重要，可以根据不同领域的需要，定制需要保存的重要内容。

# **模块二：Memory**

### 2.你理解的 Agent memory 经典框架是什么？它的发展趋势是什么，最头部的玩家在怎么做？

**我理解的memory经典框架是**：短期记忆+中期记忆（项目/会话记忆）+长期记忆+参数记忆

- 短期记忆：指的就是当前context上下文的记忆，通常直接保存最近几轮的交互内容

- 项目记忆：指的是整个长程任务中的任务目标、历史完成任务摘要等（可以按项目划分也可以按会话划分），通常作为压缩摘要一直保留在上下文中。
  对于生产类的可以包括生产环境、执行步骤约束、最佳实践等，对于情感提供类可以包括用户性格、用户偏好等。

- 长期记忆：长期记忆的定位就是跨session记录一些共性特点，与会话记忆的区别在于跨session存储、不常态化放入context、通过召回来使用。他可以保存该智能体所需要的所有外部知识、历史执行的经验等。
  这类记忆需要借助合适又强大的外挂知识库来支撑，包括但不限于RAG、GraphRAG、Wiki、纯结构化等。

- 参数记忆：这种记忆是通过预训练、微调、RLHF持久化变成模型的记忆。这类优记忆缺点明显，他不占用上下文窗口，回答自然，有泛化基础；但是更新记忆需要重新训练，容易产生幻觉或灾难性遗忘。

**发展趋势**：以上单一长短时间线的记忆维度存在缺陷，目前正在从分层记忆转向认知框架、时空记忆等。最新的论文研究记忆分为以下几种：

- token层的记忆，各种新式的外挂知识库和召回检索机制

- 模型权重层，各种微调训练框架，大家目标是实现专业方向更好的学习、非专业方向更少的遗忘。现在还存在agent能力下降等问题。

- 隐性记忆，把记忆存储在模型推理状态的内部，

- 生产记忆，模型自行生产需要的记忆

- 还有一些论文通过划分不同类别的记忆来构建认知框架，比如划分为“事实性记忆”、“经验性记忆”、“工作记忆”等。

**头部玩家：**

字节跳动的M3-agent记忆系统，实时处理视觉和听觉输入，构建更新长期记忆。他构建情节记忆、语义记忆、世界知识。[m3-agent/README.md at master · ByteDance-Seed/m3-agent](https://github.com/ByteDance-Seed/m3-agent/blob/master/README.md)

腾讯的Hy-Memory，上下文记忆：每条对话必写 L1 原始痕迹，注意力闸门判断是否值得加工，实时抽 L2 原子事实与 L3 身份画像，现场结一笔 L4 会话摘要；睡眠回放，把 System1 种下的材料长成认知：抽象 L5 心智模型、异步沉淀 L6 前瞻意图[Hy-Memory — 记住一次，随处同行](https://memory.hunyuan.tencent.com/#kernel)

一些前沿论文：Trace2Skill（轨迹局部经验蒸馏为可迁移技能）、Value-Driven ARTEM（价值驱动的记忆增强生成、Memp（程序性记忆探索）、PlugMem（从经历到知识的转化）等。他们有一些指给出方法论，有一些直接开源可复用框架。

# **模块三：Task**

### 1.对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

1、针对性模型微调训练、（我可能了解不深入），了解到的有一些自蒸馏机制、嵌套学习机制，也有一些训练框架可以做到针对知识库强化学习。
优点：就是强化训练过的内容记忆会有提高，记忆不占用上下文不因记忆增长拖慢first token速度
缺点：其他方向的内容容易灾难性遗忘，agent能力也会下降，具体表现有依赖训练框架。

2、精妙的上下文管理

- 让重要信息保留在上下文窗口内（具体可以看模块2的memory）。

- 上下文窗口限制，模型支持的最大上下文窗口通常不能达到模型的最佳表现，通过提供合理的窗口大小，适时压缩处理

- 智能记忆管理，让agent自己决定要长期记忆什么内容

3、工作流制定、任务分解、状态机管理：对于一个长程任务，可以给他制定好工作流，或者强制让agent自己定制工作流（并提供程序上的工具支持），让任务流有序推进、回滚、反思、重规划等操作。让主agent分配不同的任务给子agent，并指定不同agent之间的状态转移。

# **模块四：Tool / Session Runtime**

### 2.如果 session state 为 busy，此时用户又发来新消息，或者异步工具完成事件也到达，runtime 应该如何处理？

这个问题有点棘手。我理解中的题目背景是这样的：某一个异步操作需要很久的时间，但是用户不想等待想要继续处理，此时如何应对用户新发来的消息继续推进任务；如果继续推进，如何处理异步的工具的结果

首先这个agent解决的领域要是支持异步或存在任务是可以并行处理的，不能是类似串行或有向无环图类似的执行策略，后者任务不太依赖前者来实现。

- 想法1：我们可以为每一个对话都维护一个事件队列和状态机，状态机用于记录当前的agent运行时状态以及上下文可操作的状态；事件队列用于存储所有不能立刻返回结果的异步执行操作的结果，该队列同时可以附加优先级等策略。
  工具注册时披露告诉大模型某些工具/任务可以异步并行执行 -> llm调用异步工具时，工具立刻返回一个调用等待凭证（不含实际内容），agent状态立刻转为一个等待工具的状态，此时可以继续接收用户消息 -> 用户继续与agent交互 -> 异步工具执行结束后，结果和调用凭证放入事件队列 -> 当agent状态为空闲时主动检查事件队列是否有待处理结果，如有则召回对应凭证的上下文（如果已经被压缩）以及弹出该结果，让llm继续处理。

- 想法2：把当前被阻塞的任务分配给一个子agent进行追踪管理，主agent管理子agent的完成状态，只要主agent不依赖于子agent的结果就可以继续和用户推进。

附加：难点在于如何实现有效、高效的异步而不会导致模型产生幻觉或逻辑推理混乱；有类似于并行多agent之间相互协作的难题。

# **模块五：Agent Runtime 架构对比**

### 1.Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

Claude code（anthropic风格）的tool_use响应格式：决定工具调用时工具调用请求会放入content数组中，在这个数组里还会有普通回复的text文本等，支持并行调用、放入多个工具调用请求。工具解析时解析返回的json字符串来获取信息。

优点：

- 内容统一，一轮回复中可以交织多种回复类型（文本和工具）

- 扩展性极强，如果需要新添加各种新的内容类型，只需要新增type然后一起放入content数组即可

- 对象即用，返回内容即json字符串，可以直接解析

缺点：

- 解析需要遍历整个content数组，每一项都有可能是任意类型的消息，需要识别

- 角色语义模糊，他工具调用的返回结果用的是“user”角色

OpenAI风格的工具调用信息放在一个choices数组中，里面包含tool_calls数组，数组里是工具调用的封装对象。工具解析时解析返回对象内的参数。

优点：

- 某个类型的消息只从同一个固定路径获取，不需要像anthropic一样遍历并动态识别

- 职责清晰：assistant负责输出工具调用请求，tool负责传回结果

缺点：

- 读写解析问题：由于所有消息封装成对象，需要手动解析对象信息，历史消息存储是还涉及对象的序列化存储和反序列话读取，

- 不便于扩展：如果要加入新的类型消息，则原有字段会改变，那么旧的历史消息不能直接套用，需要兼容性调整
